# security.md — Security Decisions, Considerations & Trade-offs

Append-only log. Every security-relevant decision gets an entry at decision time, never retroactively. Format: ID | date | decision | rationale | trade-off accepted. Verification bookkeeping lives in `security-audit.md`; this file is the *why*.

---

## SEC-001 | 2026-07-16 | No evidence-document storage
**Decision:** Store cert numbers, expiration dates, and user-provided links to externally hosted evidence only. Never ingest or retain evidence documents (PDFs, certificates, transcripts beyond transient parsing).
**Rationale:** Removes the highest-sensitivity data class from custody entirely; shrinks breach blast radius, backup sensitivity, deletion obligations, and future CASA evidence scope.
**Trade-off:** Users must keep evidence elsewhere; audit-export bundles contain links, not files. Accepted — provider portals are the system of record anyway.

## SEC-002 | 2026-07-16 | Trust user-entered certifications
**Decision:** No verification of cert authenticity; all user-entered certs treated as genuine.
**Rationale:** Users have no incentive to deceive their own tracker. Verification would require handling verification identifiers and third-party checks for zero product value.
**Trade-off:** Public-profile features (post-MVP) must NOT imply verification; revisit this entry before shipping anything that displays certs to third parties.

## SEC-003 | 2026-07-16 | Gmail: on-demand one-shot scan, no persisted tokens
**Decision:** `gmail.readonly` only; user-initiated scan; token used for a single pass and discarded; extract-and-discard (message-ID + extracted fields persisted, never raw bodies); sender-scoped search queries only.
**Rationale:** Eliminates stored-token theft class entirely; minimizes restricted-scope data custody; keeps unverified-app posture viable (Testing-status ~7-day refresh expiry is irrelevant when nothing persists).
**Trade-off:** No background sync; users re-consent per scan. Accepted per D13.

## SEC-004 | 2026-07-16 | Pull-only extraction worker
**Decision:** Local Claude Code worker pulls jobs from and pushes results to the app's worker API. The server never initiates connections toward the operator's network. Worker credentials are per-worker tokens scoped to claim/submit on the staging queue only.
**Rationale:** Home network never exposed; worker compromise ≠ app compromise; token theft grants only the ability to submit staged (human-reviewed) proposals.
**Trade-off:** Worker availability is best-effort; server-side API fallback exists for steady-state.

## SEC-005 | 2026-07-16 | All extractor output is untrusted
**Decision:** Server-side JSON Schema validation on every extraction result regardless of extractor (Claude API, local Claude Code, any future local model), plus mandatory human review before publication. Provenance records extractor identity.
**Rationale:** LLM output over web-fetched content is prompt-injection-exposed by construction; validation + review bounds impact to wasted reviewer time, never corrupted canonical data.
**Trade-off:** Review queue is a labor cost; accepted as core to product trust anyway.

## SEC-006 | 2026-07-16 | Fetched/uploaded/scanned content is hostile LLM input
**Decision:** Extraction contexts get schema-constrained output, no tool access, and no access to user PII tables. Provenance metadata stored separately from content. Raw source snapshots isolated from the app DB.
**Rationale:** Prompt injection via a provider page or a crafted "certificate" upload must not be able to exfiltrate user data or trigger actions.
**Trade-off:** Extraction can't enrich against user data (fine — it shouldn't).

## SEC-007 | 2026-07-16 | Community sources are inert until Approver-triggered
**Decision:** Any user may submit URL + description; nothing is fetched or sent to an LLM until an Approver+ triggers the crawl.
**Rationale:** Prevents queue-spam from draining token budget and prevents untrusted users from steering the fetcher at arbitrary targets (SSRF-adjacent abuse, malicious pages aimed at the extractor).
**Trade-off:** Slower community-source turnaround. Accepted per D16.

## SEC-008 | 2026-07-16 | Postgres RLS beneath app-level scoping
**Decision:** Row-level security policies on all user-owned tables as a second enforcement layer under application query scoping.
**Rationale:** A single missed WHERE clause must not become a cross-tenant leak.
**Trade-off:** Policy maintenance overhead and some query-planning friction; accepted.

## SEC-009 | 2026-07-16 | Secrets & token handling baseline
**Decision:** All provider credentials and OAuth tokens envelope-encrypted at the app layer with keys in a secret manager (GCP Secret Manager per D9); nothing secret in repo, images, or plain env files; short-lived signed URLs for any object access.
**Rationale:** DB dump or backup theft must not yield usable credentials.
**Trade-off:** Key-management ceremony on a solo project; accepted as CASA-critical anyway.

## SEC-010 | 2026-07-16 | ASVS L2 alignment as build-time posture
**Decision:** Build against ASVS Level 2 chapters (authentication, session, access control, validation, crypto as the primary checklist) with continuous bookkeeping in `security-audit.md`; formal full-catalog audit is a pre-CASA milestone, but readiness is maintained continuously, not reconstructed later.
**Rationale:** Retrofitting verification evidence is the expensive part of CASA; contemporaneous records make assessment a formality. Pin the ASVS version (4.0.3 vs 5.0) at repo kickoff and record it here as SEC-011.
**Trade-off:** Ongoing bookkeeping discipline; accepted per D18 ("do things right the first time").

## SEC-012 | 2026-07-17 | Approver role confers admin review permissions in code
**Decision:** `User.has_perm`/`has_module_perms` grant approvers (role approver/admin) access to the review surfaces only — research, catalog, offers apps plus accounts.EnrollmentTask. No per-user permission rows or groups; role is the single authorization source of truth. User management stays superuser-only. Unmatched Credly badges enqueue as inert SourceSubmissions (same SEC-007 posture: user input causes no fetch or spend until an Approver triggers the crawl), deduplicated on (url, description) to bound queue-spam.
**Rationale:** Staff flag alone left approvers with zero model permissions (admin 403s, masked by a URL-only e2e assertion); scattering permission rows per user invites drift from the role field.
**Trade-off:** Permission granularity is app-level for approvers, not per-model; acceptable while the approver surface is exactly the review queue. Revisit if roles multiply.

## SEC-013 | 2026-07-17 | LLM/fetcher spend is approver-gated, categorically
**Decision:** Only approver/admin (and superuser) actions may trigger token or fetch spend — confirmed as a standing rule, not just the SEC-007 submission case. Audit at decision time: no server-side LLM call exists in the app (ANTHROPIC_API_KEY is read but unused); the only job-creating paths are the admin Trigger-crawl action (perm-gated per SEC-012) and the scheduler's re-crawl of sources that were themselves approver-promoted; the sole user-triggered server-side fetch is the Credly badges.json lookup — regex-pinned to credly.com profile URLs, no LLM involved (D6). Applies forward: the D13 Gmail "Scan my inbox" click and 5.4 upload parsing are user-facing LLM spend and MUST NOT execute directly on user action — they queue for approver-triggered or approver-batched execution unless this entry is explicitly superseded at build time.
**Rationale:** Token budget and fetcher targeting are attack surface (queue-spam, SSRF-adjacent steering, cost exhaustion); a single authorization class keeps the invariant auditable.
**Trade-off:** On-demand features become queued features (latency for users); accepted — the product's trust model already centers on human review.

## SEC-014 | 2026-07-17 | Secrets & dev cruft excluded from the Docker build context
**Decision:** Added `.dockerignore` excluding `.env`/`.env.*`, `.venv`, `node_modules`, `.git`, local DB, and caches from the image build context. The Dockerfile's `COPY . .` previously copied everything, baking `.env` (and thus SECRET_KEY, DB password, FIELD_ENCRYPTION_KEY) into image layers — a direct SEC-009 violation — plus the Windows `.venv` (broken in a Linux image) and `.git`. Also fixed `collectstatic` to run with a throwaway build-time key (not persisted as ENV) instead of `|| true`, so a `DEBUG=false` image ships a valid whitenoise manifest rather than silently-broken static.
**Rationale:** Image layers are distributable artifacts; a secret in a layer leaks on any registry push or image share, and is not removed by deleting the file in a later layer. Verified 2026-07-17 via local prod-like compose smoke test (Postgres + gunicorn + django-q, DEBUG=false): image builds clean, migrations apply on Postgres (incl. tracker.0002_rls policies), static + admin assets serve 200, worker connects.
**Trade-off:** Build context must be maintained as new secret-bearing paths appear; cheap and standard.

## SEC-015 | 2026-07-17 | Additional credential importers stay within the SEC-013 carve-out
**Decision:** Microsoft Learn and Accredible importers (apps/tracker/importers.py) make user-triggered server-side fetches, and Open Badges / LinkedIn importers parse user-uploaded files. All are the same category as the Credly lookup (SEC-013 carve-out): domain-pinned to a credential platform (learn.microsoft.com, api.accredible.com), fetching the user's OWN public credential, no LLM, no catalog write. Uploaded-file parsers NEVER fetch a hosted-assertion URL found inside the file (SSRF guard) — they reject it and ask for the .json. Only cert fields are read; PII in responses (MS Learn contactEmail/legalName/mcid) is ignored (D7). Unmatched results go to the inert SourceSubmission queue, so no importer triggers research-crawler spend (SEC-013 holds).
**Rationale:** Keeps the "no user-triggered research spend" invariant while letting users self-import from their own credential accounts; the SSRF guard prevents an uploaded file from steering the fetcher.
**Trade-off:** Name-based catalog matching for sources without stable template IDs (MS Learn, LinkedIn) yields more "no match" queue items; acceptable — misses are inert until an approver acts.

## SEC-016 | 2026-07-17 | RLS mechanism proven; production role wiring is the remaining step
**Decision:** The RLS policies (tracker/0002_rls) + the per-request `set_config('certsleuth.user_id')` middleware are verified to isolate tenants on Postgres (tests/test_rls.py, run under a non-owner role via SET LOCAL ROLE — the production model, since the table owner bypasses RLS). settings_test now honors DATABASE_URL so this runs in the CI postgres job (it was silently skipping on sqlite before, masked by `|| true`). The ICS feed is now RLS-safe: being anonymous it sets certsleuth.user_id to the token's user before querying (else the non-owner role would see 0 rows).
**Remaining for full enforcement (deploy-side, needs decisions):** (1) provision a non-owner Postgres role for the web app and point its DATABASE_URL at it (migrations/superuser stay owner); (2) run django-q workers as the OWNER role — `notify_expiring_certs` legitimately crosses all users and would see nothing under RLS; a split of web-role vs worker-role in docker-compose is the clean shape. Until (1), RLS is defense-in-depth that is present but not enforced (app connects as owner).
**Rationale:** Proving the policy mechanism now de-risks the enforcement flip; the role split is a deploy topology choice with behavior implications (background tasks, admin cross-user reads), so it is surfaced rather than silently enabled.
**Trade-off:** Two DB roles + two connection strings to manage; standard for RLS-enforced apps.

## SEC-017 | 2026-07-17 | Crawl-discovery frontier: app adds, humans process; same-domain only
**Decision:** The crawler may ADD URLs to the queue but never PROCESS them — crawl processing (fetch + LLM spend) stays human-triggered, upholding SEC-013. Links discovered while crawling a page are enqueued as inert `SourceSubmission`s with `origin=crawl` (vs `user`), deduped by canonical URL, and an approver promotes them exactly as user submissions. Discovery is deterministic (regex link scan, no LLM) so untrusted page content cannot choose crawl targets (SEC-006). **Only same-registrable-domain links are considered; cross-domain links are dropped entirely — not crawled, not queued, not recorded** — so a compromised/injected provider page can only ever point the frontier deeper into its own site, never at third parties or internal targets. Depth capped at 4 from seed, ≤50 new pages/domain/run. Full procedure: docs/crawl-procedure.md.
**Rationale:** Preserves the "no untrusted-content-triggered spend" and "no fetcher steering" invariants (SEC-006/007/013) while letting the source set grow; same-domain-only is a hard SSRF/abuse boundary that needs no allowlist maintenance.
**Trade-off:** Legitimate cross-provider references (e.g., a provider linking to a shared standard) are lost; acceptable — such sources can still be user-submitted. Barren/dead pages are parked at very low cadence, never deleted, because deletion would let rediscovery repopulate and re-crawl them.

## SEC-018 | 2026-07-18 | Deterministic cert extraction runs in-flow; output is staged like any other fact
**Decision:** `worker/extractors.py` now runs inside the worker's `fetch` pass (`_auto_facts`), emitting `certification` facts to `jobs/auto_results.jsonl` under a distinct `extractor` value (`worker-deterministic-v1`); `submit` sweeps that file alongside the operator's `results.jsonl` when no explicit path is given. Previously nothing called the module — the recovered ISACA/AWS/CompTIA/GIAC catalogs came from a one-off script, so every fresh crawl silently regressed to the generic-h1 promo-banner names those rules exist to fix. The extractor is pure stdlib regex over the snapshot: **no LLM reads the page**, so untrusted content cannot steer what is emitted (SEC-006), and no new network fetch is introduced (SEC-013 untouched — this reads a snapshot the fetch already took). Facts land in `StagedChange` for approver review exactly like Claude Code's output (SEC-005); nothing reaches the catalog unreviewed.
**Rationale:** A verified extractor that production never invokes is a latent data-quality bug. Running it at fetch time makes the fix durable, and keeping its output in a separate file preserves provenance — an approver can tell rule-derived facts from operator-derived ones, and rewriting one file never clobbers the other.
**Trade-off:** `confidence: "confirmed"` is asserted from the URL+og:title rules rather than a human read; justified because the rules are anchored to the provider's own top-level cert URL and reject slug-like, category, and promo names, and because an approver still reviews every row. Rule drift (a provider changing its og:title format) now surfaces as staged junk rather than silence — visible in review, which is the intended failure mode.

## SEC-019 | 2026-07-18 | Headless rendering by default in the worker; needs_render becomes a fact
**Decision:** The worker now renders every live 200 in headless Chromium (`worker/render.py`) rather than trying to *detect* which pages need a browser. Detection was attempted and abandoned: visible-text counts and text:markup ratios do not separate a JS shell from a content-heavy page (learn.microsoft.com/credentials/browse/ scored 0.037 against comptia's real cert page at 0.033), because site chrome and inline bundles dominate both. Two changes replace it: (1) `crawl.main_text_len()` measures content OUTSIDE nav/header/footer/aside and cookie banners — the same MS Learn page scores 175 there against 5,760–9,402 for real cert pages, a clean 30x gap; (2) `Source.NEEDS_RENDER` now means "no renderer was available for this page", a fact reported by the worker, instead of a guess inferred from markup. Conditional fetch runs first, so a 304 never reaches the browser; a render failure falls back to the raw HTML rather than dropping the page. Playwright is an OPTIONAL extra (`pip install -e .[render]`) — the server never imports it, and the worker degrades to unrendered fetches without it.
**Rationale:** Verified against the live page that motivated this: unrendered it yielded 603 visible chars and 0 links and was filed `barren` at 180-day cadence; rendered it yields 9,797 chars and 34 same-domain links, including the Microsoft renewal-policy page. The old classification was silently discarding a whole provider catalog, and no threshold tuning would have recovered it.
**Trade-off — this is the security-relevant part:** rendering EXECUTES untrusted provider JavaScript on the operator's machine, where previously page content was only ever regex-matched (SEC-006). Accepted with these bounds: it runs in Chromium's sandbox headless; this module never evaluates page-supplied script itself and returns HTML only; image/media/font requests are aborted; a 20s timeout caps hostile pages; and the same-domain frontier rule (SEC-017) still governs which URLs are ever visited. Page content still reaches no LLM and no server (SEC-005/006). Residual risk is a Chromium sandbox escape from a provider page — low, and bounded to the operator workstation, which already runs a browser against these same sites. Cost is ~6.6s/page rendered vs ~0.2s raw; bounded by conditional fetch and the tiered recrawl cadence, and `--no-render` disables it.

## SEC-020 | 2026-07-19 | Gmail scan executor: per-scan consent, token confined to one request, metadata only
**Decision:** Implemented the `run_scan` body (apps/tracker/gmail.py + gmail_scan_run/callback views). Flow: an APPROVED GmailScanRequest (SEC-013 gate unchanged) lets the user start a Google OAuth consent round-trip; the callback exchanges the code and uses the access token for exactly one bounded pass inside that request's frame, then discards it — `access_type=online` means no refresh token ever exists, and no model, session, or log receives the token. The `state` parameter is a signed (user, scan) pair with a 10-minute max age, so a callback cannot be replayed across users or scans. The Gmail query is pinned to known issuer senders (never the whole mailbox), capped at 50 messages, and requests `format=metadata` only — message BODIES are never fetched from Google at all, which is stronger than fetch-and-discard. Subject-line parsing is deterministic regex (no LLM reads mail); parsed sightings flow through the shared importer preview/confirm (user confirms rows into their own tracker; unmatched names queue inert for research, D16). The scaffold's earlier note that output goes to StagedChange was corrected: scan findings are user-credential sightings, not catalog facts, so the importer path (SEC-015 family) is the right boundary.
**Rationale:** One-shot consent + online-only tokens + metadata-only fetches minimize both the stored-secret surface (nothing to encrypt because nothing persists) and the data-exposure surface (no body content ever transits our server). Restricted-scope posture: works for console-registered test users now; public rollout requires Google's CASA verification (accepted cost per spec 5.1).
**Trade-off:** Metadata-only parsing misses certs mentioned only in bodies, and unparseable subjects are dropped rather than guessed — lower recall, zero fabrication. Recall can grow later via per-issuer subject templates or the 5.4 LLM-upload path, both without changing this boundary.

## SEC-021 | 2026-07-20 | Login brute-force lockout (django-axes), IP-based
**Decision:** Added `django-axes` to gate the login view, which was internet-facing with no throttle once production went live (certsleuth.com). Config (`config/settings.py`): `AxesStandaloneBackend` first in `AUTHENTICATION_BACKENDS` — it only performs the lockout check and defers credential verification to `ModelBackend`; `AxesMiddleware` last in the stack turns a lockout into the response. Lockout is **IP-based** (`AXES_LOCKOUT_PARAMETERS = ["ip_address"]`), 5 failures, 1-hour cool-off, reset-on-success, returning a friendly 429 (`registration/lockout.html`). The database handler is used (`AxesDatabaseHandler`) so lockout state is shared across the two gunicorn workers without adding Redis to the 1 GB VM. Two environment-specific fixes: (1) `AXES_USERNAME_FORM_FIELD = "username"` — Django's `AuthenticationForm` submits the identifier under the literal key `username` even though our `USERNAME_FIELD` is `email`, and axes otherwise defaults to `USERNAME_FIELD`, reading the wrong key and recording `username=None`; pinning it captures the targeted account on each `AccessAttempt` for approver forensics. (2) Behind Caddy, `AXES_IPWARE_PROXY_COUNT = 1` when `not DEBUG`, so the resolved client IP is the address the trusted proxy appended, not a client-spoofable `X-Forwarded-For` value. Test: `tests/test_login_lockout.py` proves the limit-th failure locks and a subsequent *correct* password is still refused (attempt-based, not credential-based).
**Rationale:** IP-based lockout is DoS-safe — an attacker cannot lock a victim's account, only their own source IP — and it also catches credential-stuffing that spreads a few guesses across many accounts from one IP (which a per-(account,IP) lock would miss). Username-based locking was deliberately avoided precisely because it hands an attacker an account-lockout DoS.
**Trade-off:** IP lockout can collaterally block legitimate users who share a NAT/corporate egress IP with an attacker; bounded by the 1-hour cool-off and acceptable at invite-only scale. Distributed (many-IP) brute force is not stopped by IP lockout alone — the Argon2 hashing + 12-char minimum + common-password validator (SEC-009) remain the standing backstop, and TOTP stays an open option (ASVS V2).
